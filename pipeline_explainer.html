A complete technical explanation of how the pipeline was built, how it works, how it is deployed, and why it was moved from Vercel to PythonAnywhere — explained from scratch.

1 What Does This Pipeline Do?

Start here if you know nothing about this project.

The Problem It Solves

Collegedunia has 8,000+ college pages. Each college page has multiple article sections — Admissions, Fees, Placements, Scholarships, etc. We want to know: for each college article, where does it rank on Google?

For example: if a user searches "IIT Bombay B.Tech Fees", does the Collegedunia page rank #1 on Google, or is it at position #8? If it's at #8, we need to improve the article.

Checking rankings manually for 8,000+ colleges × 5+ keywords = 40,000+ Google searches — impossible to do by hand. This pipeline automates all of it.

🎯 Input

A Google Sheet with all college names, IDs, course names — the Source tab.

⚙️ Process

Automatically build search keywords, query Google rankings via Bright Data API, and store results.

📊 Output

A Google Sheet showing where each Collegedunia article ranks on Google, updated every 15 days.

4 The 3 Pipeline Stages — Step by Step

When you run python run_pipeline.py, it executes 3 stages in sequence.

1
Stage 1 — phase1_populate.py
Source tab → Intermediate tab (prepares keywords for ranking)

This script decides which colleges to rank today and builds search keywords for each one. Here is exactly what it does, step by step:

Step A — Load the checkpoint

It reads GS_Pipeline_State tab to find out where in the Source list it left off last time (checkpoint_row). This is like a bookmark — so it doesn't start from the top every day.

Step B — Select colleges using 40% batch sampling

It takes 50 colleges at a time from Source (starting at the checkpoint). From each group of 50, it picks the top 40% = 20 colleges. It prefers colleges with higher search traffic (from the Content tab). It keeps picking until it has 550 colleges for today, or runs out of Source rows.

Step C — Skip colleges ranked recently (15-day cycle)

If a college was already ranked within the last 15 days, it is skipped. There is no point checking its ranking again so soon.

Step D — Build keywords for each selected college

For each college, it creates 5–6 search keywords based on a short name (from Colleges_Short_Form.csv) and course name:

IIT Bombay B.Tech CSE IIT Bombay B.Tech CSE Admissions IIT Bombay B.Tech CSE Fees IIT Bombay B.Tech CSE Placements IIT Bombay B.Tech CSE Scholarships IIT Bombay B.Tech CSE (Computer Science)

Each keyword maps to one article "silo" on Collegedunia — Main page, Admissions article, Fees article, etc.

Step E — Write to Intermediate tab

All keywords are appended as rows in the Intermediate tab (columns A–E). Columns F, G, H are left blank — Apps Script will fill those in during Stage 2.

Step F — Update Final tab identity rows

Each selected college gets an identity row in the Final tab (columns A–E). The 13 result columns for this run are added as blank headers, ready for Stage 3 to fill in.

Step G — Save checkpoint + write run_start_col to Source!Z1

The new checkpoint position is saved back to the GS_Pipeline_State sheet. The current run's starting column number is written to Source!Z1 so Apps Script knows where to write results.

⚠️
Daily guard: Phase 1 only runs once per calendar day. If it already ran today, it exits immediately (exit code 2) and the pipeline stops gracefully.
2
Stage 2 — Apps Script (intermediate_apps_script.js)
Intermediate tab → Google SERP rankings via Bright Data API

This is where the actual Google ranking check happens. The Python script calls checkRanksBatched() inside Google Apps Script via the Apps Script Execution API.

Why Apps Script for this step?

Google Apps Script runs inside Google's infrastructure and can read/write Google Sheets directly without API quotas. It also supports UrlFetchApp to call Bright Data. Running this from Python would require far more API calls and authentication complexity.

What checkRanksBatched() does:

1. Opens the Intermediate sheet and finds all rows where column E (keyword) has a value but column F (Rank) is empty — these are unranked rows.

2. Takes them in batches of 10 at a time and calls the Bright Data SERP API simultaneously (using UrlFetchApp.fetchAll for parallelism).

3. For each keyword, Bright Data returns the top 50 Google search results. The script scans them to find any URL containing collegedunia.com and records its position number (rank).

4. It writes the rank, the found URL, and a timestamp into columns F, G, H of that row in Intermediate.

5. Apps Script has a 6-minute execution limit. After 5.5 minutes it stops — even if rows remain. Python then calls it again in a new round.

How Python manages the rounds:

Python calls Apps Script, waits 15 seconds, then counts unranked rows. If rows remain, it calls Apps Script again. This repeats up to 70 rounds (~7 hours total if needed for 40,000+ rows). Each round covers ~200–500 keywords.

🔄
Parallel fetching: By firing 10 keywords simultaneously per fetchAll call, Stage 2 is ~10× faster than sequential. 550 colleges × 5 keywords = 2,750 keywords → ~6 rounds (~35 minutes).
3
Stage 3 — phase2_build_master.py
Intermediate tab → Final / Content tab (stores results permanently)

Once all keywords are ranked in Stage 2, Phase 2 reads the ranked Intermediate rows and writes results permanently into the Final and Content tabs.

What it does:

1. Reads all ranked rows from Intermediate (rows that now have values in columns F, G, H).

2. Groups them by college–course pair. Each pair has up to 6 silos (Admissions, Fees, Placements, Scholarships, Main, Single_Course).

3. Writes a date-stamped column group (13 columns) into the Final tab for the current run. Each silo gets its rank + URL + timestamp.

4. The silo type is detected automatically from the keyword suffix — "IIT Bombay B.Tech CSE Fees" → Fees silo, "IIT Bombay B.Tech CSE Admissions" → Admissions silo, etc.

✅
After Stage 3: The pipeline is complete. The Final tab now has a new set of 13 columns showing today's ranks for all processed colleges. The pipeline exits successfully.

5 The 15-Day Cycle — How Rotation Works

The pipeline doesn't re-rank the same colleges every day — it rotates through all 8,000+ colleges over 15 days.

The Problem with 8,000 Colleges

You can't rank 8,000 colleges every day — that would be 40,000+ API calls daily. Instead, the pipeline takes 550 colleges per day and spreads all 8,000 across ~15 days. Each college gets re-ranked once every 15 days, which is frequent enough to detect ranking changes.

How the Checkpoint Works

Think of Source as a circular queue. The pipeline keeps a checkpoint_row that remembers its position in Source:

Day 1 — Run 1
Starts at row 0. Takes batches of 50, picks top 40% (=20). Collects 550 colleges. Saves checkpoint at row ~1,375. These 550 colleges are ranked today.
Day 2 — Run 1 continues
Resumes from row 1,375. Picks next 550 colleges. They haven't been ranked yet (or were ranked >15 days ago), so they're due.
Days 3–14 — Middle of cycle
Pipeline runs daily. Each run advances the checkpoint further through Source. Colleges already ranked recently are skipped.
Day 15 — Checkpoint wraps
Checkpoint reaches the end of Source (~8,000 rows), wraps back to 0. A new 15-day cycle begins. Day 1 colleges are now 15 days old → due for re-ranking.
⚠️
15-day skip rule: If a college was ranked less than 15 days ago, it is skipped even if the checkpoint reaches it. This prevents double-ranking and ensures each college is ranked exactly once per cycle.

New College Handling

When a college is added to the Source tab for the first time: it is ranked the next time the checkpoint reaches its row. If the day's 550-college limit is already full, it is added to a deferred queue in GS_Pipeline_State and ranked the next day as a priority.

Deletion Sync

If a college is removed from the Source tab, Phase 1 automatically removes all its rows from the Intermediate and Final tabs too. This keeps all three tabs in sync.

7 Why Vercel Failed — Why PythonAnywhere Works

This is one of the most important technical decisions in this project.

❌ Vercel — Why It Didn't Work

  • Serverless functions have a hard timeout. Vercel functions run for max 10–60 seconds (depending on plan). The pipeline needs to run for 30 minutes to several hours. It would get killed halfway through.
  • No persistent filesystem. Vercel functions are stateless. Files like token.json and credentials.json cannot be stored permanently — they disappear after each function execution. Every run would need fresh OAuth login, which requires a browser.
  • Can't manage long-running processes. The Apps Script loop (up to 70 rounds, 5.5 min each) is a long-running process. Vercel cannot keep a function alive for that long.
  • No cron for long tasks. Vercel Cron Jobs also run as serverless functions — same 10–60 second limit.
  • No SSH or manual intervention. If something breaks, you can't SSH into Vercel to inspect files, re-authenticate, or debug.
  • Memory limits. Loading 8,000 rows of spreadsheet data into memory may exceed Vercel's function memory cap.

✅ PythonAnywhere — Why It Works

  • Always-on Linux server. Processes can run for hours. No timeout for background scripts.
  • Persistent filesystem. token.json, credentials.json, pipeline_state.json, log files — all stored permanently on disk. OAuth tokens survive across runs.
  • Full Python environment. Install any library (gspread, flask, google-auth, etc.) once and it stays installed.
  • SSH access. You can log in via SSH or the in-browser console to debug, re-authenticate, or manually trigger runs.
  • Flask web app support. The dashboard (app.py) runs as a proper web server, not just a serverless function.
  • Background tasks. PythonAnywhere supports long-running background scripts through Scheduled Tasks or Always-On tasks.
  • Free tier is sufficient. For this pipeline's needs, the PythonAnywhere free/basic tier provides enough compute time and storage.
📌
The core principle: Vercel is designed for web apps and APIs that respond in milliseconds. This pipeline is a batch processing job that runs for hours. These are fundamentally different use cases. PythonAnywhere is essentially a VPS (virtual private server) that is always running — the correct tool for this job.

10 Pros, Cons, and Limitations

An honest assessment of the current architecture.

✅ Strengths

  • Fully automated end-to-end — runs itself every 15 days with zero manual intervention
  • Scalable to 40,000+ keywords — the round-based Apps Script loop handles any volume
  • No data loss — GS_Pipeline_State ensures the checkpoint survives server restarts and GitHub Actions' stateless environment
  • Self-healing — pipeline lock clears on crash; daily guard prevents double-runs
  • Historical tracking — Final tab grows rightward, enabling ranking trend analysis over time
  • Smart prioritisation — high-traffic colleges (from Content tab) are ranked first
  • Deletion sync — removing a college from Source automatically cleans up Intermediate and Final
  • Live dashboard — Flask UI provides visibility without needing to SSH into the server
  • Cost-effective — PythonAnywhere free/basic tier + Google Sheets + GitHub Actions (free) = near-zero cost

⚠️ Limitations

  • Single point of failure on PythonAnywhere — if the server goes down mid-run, the pipeline stops; manual restart required
  • Apps Script 6-minute limit — each invocation covers limited rows; very large batches need many rounds
  • Bright Data API cost — SERP API calls are paid; high volume = higher cost
  • Google API rate limits — gspread calls can hit Sheets API quotas on very large sheets
  • OAuth token expiry — token.json refresh tokens can expire; requires manual re-authentication via browser
  • No parallel college processing — colleges are ranked sequentially by Apps Script, not in parallel across multiple workers
  • Final sheet column explosion — after 50+ runs, the Final tab has 650+ columns, which can be hard to navigate
  • No alerting on failure — if a run fails silently, you only notice when checking the dashboard

11 Keyword Generation Logic — The "Silo" Concept

How each college's keywords are derived and what "silo" means.

What is a Silo?

On Collegedunia, each college has multiple article pages — the main overview page, an Admissions article, a Fees article, a Placements article, etc. We call each article type a "silo." The pipeline tracks the Google ranking for each silo separately.

The Short Name

Full college names like "Indian Institute of Technology Bombay, Powai, Maharashtra" make terrible search queries. The pipeline uses a short form from Colleges_Short_Form.csv (e.g., "IIT Bombay"). If a college isn't in the CSV, it auto-extracts the short form from the full name.

Keyword Formula

The base keyword = [College Short Name] + [Course Short Name]. Then one keyword per silo is created by appending a suffix:

SiloSuffix AddedExample KeywordMaps to which article
MainnoneIIT Bombay B.Tech CSECollege overview / main page
Admissions" Admissions"IIT Bombay B.Tech CSE AdmissionsAdmissions article
Fees" Fees"IIT Bombay B.Tech CSE FeesFees article
Placements" Placements"IIT Bombay B.Tech CSE PlacementsPlacements article
Scholarships" Scholarships"IIT Bombay B.Tech CSE ScholarshipsScholarships article
Single_Course" (specialization)"IIT Bombay B.Tech CSE (Computer Science)Specific course page

The Single_Course silo only exists if the course name has a specialization in parentheses, e.g., "B.Tech (Computer Science and Engineering)".

12 Glossary — Key Terms Explained

SERP

Search Engine Results Page — the Google results page you see when you search for something. The pipeline checks what position Collegedunia appears on this page.

Bright Data

A data platform that provides SERP scraping — it queries Google on your behalf and returns structured results. Used here because direct Google scraping is blocked.

Apps Script

Google's JavaScript runtime that runs inside Google Workspace (Sheets, Docs, etc.). It can read/write Sheets and make HTTP requests natively, without complex auth setup.

OAuth / token.json

Google's authentication protocol. After you log in once via browser, Google gives you a token.json file that the pipeline uses to access Sheets without logging in again.

Checkpoint Row

A number saved in GS_Pipeline_State that remembers where in the 8,000-row Source list the pipeline stopped last time. Like a bookmark in a book.

Silo

One specific article type on a college's Collegedunia page — Main, Admissions, Fees, Placements, Scholarships, or Single_Course. Each has its own Google ranking.

Pipeline Lock

A flag set in pipeline_state.json while the full pipeline is running. Prevents Quick Run from starting simultaneously and corrupting data.

Run Number

Which iteration of the 15-day cycle this is. Run 1 = first ever ranking; Run 2 = second ranking (15 days later), etc. Each run adds 13 columns to Final.
